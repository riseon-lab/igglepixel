# ── Runtime-installed Python deps ──────────────────────────────────────────
# docker/entrypoint.sh pip-installs this file on every pod boot if present.
# Keep packages here that are feature-flag deps or that we want to be able
# to update without rebuilding the Docker image. Anything that breaks the
# image base (torch, CUDA-pinned packages, etc.) belongs in the Dockerfile.
#
# Why peft is here, not in the image:
#   diffusers >= 0.21 requires peft for any LoRA loading / set_adapters call.
#   Without it, pipe.load_lora_weights raises "PEFT backend is required for
#   this method." Adding it via runtime install gets the fix to live pods
#   without the 25-min image rebuild + push cycle for new CUDA bases.
#
# Why ftfy:
#   Several text encoders (CLIP, Qwen2.5-VL preprocessing, etc.) call
#   ftfy.fix_text() on prompts to normalise unicode. Without it generation
#   fails with "NameError: name 'ftfy' is not defined".
#
# Why imageio + imageio-ffmpeg:
#   Saving video frames to mp4. imageio is the path diffusers' export_to_video
#   prefers; it shells out to ffmpeg (already in the image) with libx264 +
#   yuv420p + +faststart. The cv2 backend it would fall back to without
#   imageio silently produces non-zero but invalid mp4s (missing/bad moov,
#   no faststart, fps mismatch) — videos that "exist" but show 0 seconds.
#
# Why diffusers==0.38.0:
#   Qwen/Qwen-Image-Edit-2511 uses QwenImageEditPlusPipeline. Older image
#   builds may have the original Qwen edit pipeline but not the 2511-ready
#   Plus loader. Qwen/Qwen-Image-2512 also expects a current QwenImage
#   pipeline implementation. FLUX.2 [klein] uses Flux2KleinPipeline,
#   Z-Image Turbo uses ZImagePipeline/ZImageImg2ImgPipeline, LongCat uses
#   LongCatImagePipeline/LongCatImageEditPipeline, and InstantX Qwen
#   ControlNet uses QwenImageControlNetPipeline/QwenImageControlNetInpaintPipeline.
#   HiDream uses HiDreamImagePipeline. SenseNova-U1 is intentionally isolated
#   in a runtime profile because it pins torch/Python more aggressively. Keep
#   this pinned so LoRA loading semantics cannot drift between pod restarts.
#
# Why peft==0.17.1:
#   PEFT is the backend Diffusers uses for LoRA adapter loading and activation.
#   Keep it pinned with Diffusers so the same LoRA does not see a different
#   loader implementation after a fresh runtime install.
#
# Why uv + virtualenv:
#   Per-runtime profiles (registry `runtime_profiles` entries) need isolated
#   venvs when a model brings deps that conflict with the shared image. Some
#   pod images do not have the entrypoint-installed uv binary or a working
#   `python -m venv` / ensurepip path, so the backend prefers uv and falls
#   back to virtualenv before giving up.
#
# Why bitsandbytes>=0.46:
#   Blackwell (RTX 5090, sm_120) requires bnb 0.46+ for working int8
#   kernels. Older versions (<0.45) raise `AttributeError: ... getCurrentStream`
#   from the int8 functional path because they call into a CUDA stream
#   context that doesn't exist on sm_120. BF16 doesn't go through bnb so it
#   masks the issue — only int8/NF4 paths break.
#
# Why torchao==0.15.0:
#   Qwen text-to-image INT8 uses Diffusers' TorchAO quantization backend by
#   default. It avoids bitsandbytes CUDA stream failures seen on some L40S pod
#   images while preserving the lower-VRAM INT8 path. Diffusers 0.38.0's
#   TorchAoConfig requires torchao >= 0.15.0 to function properly, while
#   transformers 4.49.0 relies on legacy functional APIs (like int4_weight_only)
#   that were removed in torchao 0.16.0. Pinning exactly to 0.15.0 satisfies both.
#
# Why torch/torchvision/torchaudio pins here:
#   Keep the shared backend Python on a single CUDA 12.8 PyTorch stack so
#   post-hoc installs cannot drift torch and torchaudio onto different CUDA
#   builds. Qwen/vLLM vision is intentionally isolated in its runtime profile;
#   do not install vLLM into the backend manager process from this file.
