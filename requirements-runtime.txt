# ── Runtime-installed Python deps ──────────────────────────────────────────
# docker/entrypoint.sh pip-installs this file on every pod boot if present.
# Keep packages here that are feature-flag deps or that we want to be able
# to update without rebuilding the Docker image. Anything that breaks the
# image base (torch, CUDA-pinned packages, etc.) belongs in the Dockerfile.
#
# Why peft is here, not in the image:
#   diffusers >= 0.21 requires peft for any LoRA loading / set_adapters call.
#   Without it, pipe.load_lora_weights raises "PEFT backend is required for
#   this method." Adding it via runtime install gets the fix to live pods
#   without the 25-min image rebuild + push cycle for new CUDA bases.
#
# Why ftfy:
#   Several text encoders (CLIP, Qwen2.5-VL preprocessing, etc.) call
#   ftfy.fix_text() on prompts to normalise unicode. Without it generation
#   fails with "NameError: name 'ftfy' is not defined".
#
# Why imageio + imageio-ffmpeg:
#   Saving video frames to mp4. imageio is the path diffusers' export_to_video
#   prefers; it shells out to ffmpeg (already in the image) with libx264 +
#   yuv420p + +faststart. The cv2 backend it would fall back to without
#   imageio silently produces non-zero but invalid mp4s (missing/bad moov,
#   no faststart, fps mismatch) — videos that "exist" but show 0 seconds.
#
# Why diffusers>=0.37.1:
#   Qwen/Qwen-Image-Edit-2511 uses QwenImageEditPlusPipeline. Older image
#   builds may have the original Qwen edit pipeline but not the 2511-ready
#   Plus loader. Qwen/Qwen-Image-2512 also expects a current QwenImage
#   pipeline implementation.
#
# Why uv + virtualenv:
#   Per-runtime profiles (registry `runtime_profiles` entries) need isolated
#   venvs when a model brings deps that conflict with the shared image. Some
#   pod images do not have the entrypoint-installed uv binary or a working
#   `python -m venv` / ensurepip path, so the backend prefers uv and falls
#   back to virtualenv before giving up.
